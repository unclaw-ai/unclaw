# Vision

## What Unclaw is trying to become

Unclaw is trying to become a **lightweight, secure, privacy-first, local-only autonomous AI agent runtime** for real local models and real consumer hardware.

The goal is not to build another cloud-dependent assistant.
The goal is not to build a giant framework.
The goal is not to fake autonomy through slash commands and prompt tricks.

The goal is to build a local assistant that:
- feels natural to use,
- stays fast on modest hardware,
- can go deeper when needed,
- can use tools intelligently,
- remains transparent,
- and remains understandable to developers.

---

## The product promise

Unclaw should eventually feel like a real personal assistant that lives on your own machine.

That means:
- local models by default,
- local memory by default,
- local tools by default,
- local data control by default,
- no mandatory cloud lock-in,
- and no recurring AI bill just to get the core experience.

---

## The truth we must keep in mind

A local assistant is not automatically an agent.
A model with tools is not automatically an agent.
A chatbot with slash commands is not automatically an agent.

Unclaw only deserves the word **agent runtime** if it can actually:
- understand a request,
- decide whether tools are needed,
- use those tools autonomously,
- observe the results,
- continue in bounded steps,
- and return a natural final answer.

This distinction must remain central to the project.

---

## Long-term ambition

Unclaw aims to become a reference project for local AI agents by being:
- cleaner than heavyweight agent stacks,
- safer than reckless automation-first demos,
- more practical than academic prototypes,
- more understandable than opaque orchestration frameworks,
- and more usable than local projects that expose too much internal machinery.

The ambition is high:
Unclaw should become one of the most convincing local-first autonomous agent runtimes for real users and real machines.

---

## Core product principles

### Local-first
The core of the project must work without cloud services.
Local models, local memory, local tools, and local execution are the true default.

### Privacy-first
Data should stay local by default.
Users should be able to understand what is stored, where, and why.

### Security-first
Autonomy is only useful if it is safe.
The product must treat fetched content, local paths, permissions, and risky actions conservatively.

### Lightweight by design
The project must stay lean:
- few dependencies,
- bounded logic,
- low overhead,
- no unnecessary framework gravity,
- and no architecture that assumes giant models or enterprise infrastructure.

### Adaptive intelligence
Unclaw should not treat every request the same way.
It should adapt routing, context, tool use, and depth to the task.

### Fast and deep dual behavior
The product must support:
- a fast path for simple requests,
- and a deeper bounded path for more complex tasks.

### Transparency
Users and developers should be able to inspect:
- routing,
- model choice,
- tool use,
- memory selection,
- timing,
- execution steps,
- and failures.

### Clean extensibility
The codebase should remain readable and extensible as new tools, channels, memory layers, and providers are added.

---

## UX vision

The best version of Unclaw should feel simple.

A normal user should be able to ask naturally for:
- information,
- file help,
- research,
- structured summaries,
- and multi-step assistance,
without needing to know which internal tool is being used.

The user experience should feel:
- direct,
- professional,
- clear,
- fast,
- and trustworthy.

Internal mechanics should remain visible through logs and traces for those who want them, but they should not pollute the normal answer style.

---

## What Unclaw must not become

Unclaw must not become:
- a cloud SaaS product,
- a thin wrapper over remote APIs,
- a monolithic chatbot with prompt stuffing,
- a brittle keyword-trigger machine,
- a framework-heavy pile of abstractions,
- an opaque autonomous system that hides what it is doing,
- or a manual-tool chatbot sold as a serious agent.

---

## Target users

The main target users are:
- developers,
- makers,
- local AI enthusiasts,
- power users,
- privacy-conscious users,
- and later, broader users who want a practical assistant without cloud dependence.

The project must keep enough engineering rigor for experts,
while moving toward a UX that does not require expert behavior from the user.

---

## Success criteria

A successful Unclaw project should:
- run well on modest local hardware,
- feel responsive,
- remain understandable to developers,
- support multiple local model sizes,
- use tools intelligently,
- keep execution bounded and safe,
- provide clear logs and traces,
- keep data local,
- remain lightweight,
- and offer a strong base for future growth.

The real success condition is not “has many tools”.
The real success condition is:
**it behaves like a trustworthy local autonomous assistant while staying lightweight, secure, and understandable.**
