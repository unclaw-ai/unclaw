==================================================
PRODUCT CONTEXT, OBSERVED ISSUES, AND FUTURE DIRECTION
==================================================

This section provides additional human context for the audit.
It is NOT a source of truth by itself.
The auditor must verify, refute, refine, or confirm each point against the actual codebase.

The goal is to help the audit understand:
- real-world issues observed during usage
- architectural improvement ideas
- the intended future direction of Unclaw
- the desired differentiation versus OpenClaw

==================================================
1. STRATEGIC CONTEXT
==================================================

OpenClaw has recently improved local-model support significantly, including official Ollama integration.
Because of this, Unclaw should NOT be evaluated only as:
- “OpenClaw but local”
- or “OpenClaw with Ollama”

That differentiation is no longer strong enough.

The audit must evaluate whether Unclaw’s strongest future positioning is instead something closer to:

- ultra-simple local-first desktop agent
- mono-user, privacy-first runtime
- highly optimized for small and medium local open-source models
- useful for concrete local tasks on personal machines
- explicit, understandable, permissioned local actions
- lighter, simpler, and more controllable than broader platform-style agent systems

The key strategic question is:

**What can Unclaw become that OpenClaw is less likely to do especially well?**

Strong candidate directions to evaluate:
- aggressive simplicity
- small-model-first design
- dynamic context minimization
- memory as optional retrieval instead of heavy prompt stuffing
- strong Ollama-specific optimization
- desktop-local task usefulness
- understandable security and permissions
- clean public UX for normal users

==================================================
2. LONG-TERM PRODUCT AMBITION
==================================================

The project should eventually become a true local-first agent runtime that is:
- simple to install
- simple to understand
- useful on consumer hardware
- reliable
- privacy-first
- secure-first
- cloud-optional or cloud-free for core intelligence
- capable of doing more than chatting and web search

At a minimum, the long-term product should evolve toward useful local action capability such as:
- web research
- system information awareness
- note creation / reading / editing
- file management
- document reading / editing
- simple local automations
- safe terminal actions
- basic desktop actions
- useful memory and continuity

The audit must determine:
- how far the current codebase is from that
- which parts of this are realistic near-term V1 goals
- which parts are too early
- which parts are true differentiators versus feature noise

==================================================
3. IMPORTANT ARCHITECTURAL IDEAS TO EVALUATE
==================================================

These are not confirmed design decisions.
The audit must judge whether they are technically sound, realistic, and worth prioritizing.

A. Dynamic context per turn
The runtime should avoid injecting large amounts of unnecessary context on every request.
Example intuition:
for a simple request like weather, the model may only need:
- system prompt
- current date/time/location
- small session context if needed
- relevant tool output

Not full history, not full summary, not old irrelevant search traces.

The audit should evaluate whether Unclaw could benefit from:
- per-turn context planning
- context budgeting
- selective history inclusion
- selective memory retrieval
- lighter prompts for simple tasks

B. Memory as optional retrieval, not giant default injection
Long-term, memory should probably become something more like a retrievable source/tool:
- session memory
- long-term memory
- project memory
- notes
- task state

The runtime should decide when memory is needed, and which memory to consult, rather than always stuffing large memory blocks into context.

The audit should evaluate whether current architecture supports this direction.

C. Small-model efficiency as a first-class principle
Unclaw should ideally become unusually good on:
- small models
- moderate RAM machines
- consumer hardware
- low-latency local usage

This means the audit should explicitly examine:
- context size discipline
- prompt budget discipline
- memory injection cost
- routing efficiency
- unnecessary tool/model calls
- model warm/loading behavior

D. Public desktop product potential
Long-term public distribution may require:
- desktop GUI
- easy install
- eventually Windows packaging / executable distribution
- smoother onboarding
- simple permission UX
- clear tool visibility

The audit should judge whether the current architecture can evolve toward this without a total rewrite.

==================================================
4. OBSERVED PROBLEMS TO VERIFY
==================================================

The following are manually observed issues during usage.
They must be treated as hypotheses to investigate, not as guaranteed truths.

----------------------------------
PROBLEM A — Model lifecycle / warm-loading / unload behavior
----------------------------------

Observed symptom:
- the model appears to unload from RAM after each answer and reload on the next prompt
- startup does not obviously preload the selected model
- switching models does not obviously warm-load the new model
- there may be no user-facing control for “keep model loaded” vs “unload after inactivity”

Questions for the audit:
- what is actually controlled by Unclaw?
- what is controlled by Ollama?
- is keep_alive implemented correctly?
- are startup and model-switch paths really warming models?
- should there be explicit user control for this?

----------------------------------
PROBLEM B — Web tools package structure may be confusing
----------------------------------

Observed symptom:
- there is a web-related package/folder structure that may not fully match where the real web modules live
- the current layout may be intentional, transitional, or confusing

Questions for the audit:
- is the current web-tools structure clean and future-proof?
- is it misleading?
- is it real technical debt?
- should web internals be reorganized more clearly?

----------------------------------
PROBLEM C — Channel package structure may not scale cleanly
----------------------------------

Observed symptom:
- Telegram-related files may be too flat under channels/
- future support for more messaging platforms may make this structure awkward

Questions for the audit:
- is the current structure acceptable for MVP?
- does it become a blocker for multi-channel evolution?
- should channel implementations move toward subpackages?

----------------------------------
PROBLEM D — Search / research results feel too narrow or too lossy
----------------------------------

Observed symptom:
- search-backed answers often feel based on too few sources
- important information is often missed
- follow-up turns sometimes fail to recover missing details
- synthesis may be too shallow
- retrieval may be too narrow or fragile

Example class of failure observed:
- person lookup can initially fail
- search can return partial truth
- follow-up questions can still remain weak or inconsistent

Questions for the audit:
- is the bottleneck in query generation?
- retrieval breadth?
- ranking?
- source filtering?
- extraction?
- synthesis?
- context budget?
- model weakness?
- prompt design?
- routing?
- follow-up grounding?
- or a combination?

----------------------------------
PROBLEM E — Search answer length often feels too weak
----------------------------------

Observed symptom:
- answers for web-backed requests are often too short
- short replies are sometimes fine for tiny factual questions
- but for richer news/research questions, the result often feels underdeveloped

Questions for the audit:
- is this mostly due to model size?
- current output instructions?
- synthesis design?
- token/output budgeting?
- search result quality?
- answer shaping policy?

----------------------------------
PROBLEM F — Stream interruption fallback appears in UX
----------------------------------

Observed symptom:
- UI sometimes displays a fallback message such as stream interruption before showing saved final reply

Questions for the audit:
- is this expected fallback behavior?
- is it a UX issue only?
- does it reveal deeper runtime/streaming fragility?
- is it still present in the current codebase?

----------------------------------
PROBLEM G — Some routing / runtime logic may still rely too much on fixed lists or deterministic shortcuts
----------------------------------

Observed symptom:
- parts of the code may still use manual lists / hardcoded heuristics in ways that reduce true agentic behavior
- risk: system behaves like a rigid process follower rather than an intelligent runtime

Questions for the audit:
- where does this still exist?
- is it acceptable MVP pragmatism?
- or does it block scalability / multilingual behavior / generalization?

----------------------------------
PROBLEM H — Memory still feels limited, brittle, or sometimes false
----------------------------------

Observed symptom:
- the memory system is improved, but still sometimes fails to recover important earlier facts
- it may preserve partial continuity while missing key details
- it can produce incorrect recall of earlier conversation state

Questions for the audit:
- how reliable is current session memory?
- how summary-driven is it?
- is summary reinjection strong enough?
- is structured memory now actually useful internally?
- is memory safe but too weak, or dangerous because it sounds more reliable than it is?

----------------------------------
PROBLEM I — Product usefulness is still too narrow
----------------------------------

Observed symptom:
- today the system can discuss, search the web, and maintain some continuity
- but it still may not feel broadly useful as a local agent for normal users

Questions for the audit:
- what capability families are still missing for a credible public V1?
- what minimum local action surface is required?
- which missing tools matter most?

5. AREAS OF PRODUCT IMPROVEMENT TO EVALUATE
==================================================

The audit must evaluate the value, urgency, and realism of the following possible improvements.

A. System information tools
Examples:
- OS
- language
- timezone
- date/time
- hostname
- basic environment awareness

Why it matters:
A local agent that does not understand the local environment is heavily limited.

B. Notes capability
Examples:
- create note
- read note
- update note
- list notes
- simple organization

Why it matters:
This is likely one of the highest-value low-complexity local tools.

C. File management tools
Examples:
- list
- search
- move
- copy
- rename
- delete with confirmation
- sort / classify basic folders

Why it matters:
This is a key shift from “chat assistant” to “useful local agent”.

D. Document tools
Examples:
- read txt / md / json / csv first
- later pdf / docx
- summarize
- compare
- extract facts
- edit simple content

E. Safe terminal actions
Examples:
- allowlisted or permission-gated shell actions
- preview before execute
- timeouts
- cancellation
- explicit user approval

Why it matters:
Very high utility for technical users and automations.

F. Better multi-OS support
Especially:
- macOS UX differences
- onboarding behavior
- terminal interaction quality
- shell/path/platform assumptions

G. MLX / Apple-local optimization
Audit should judge whether supporting MLX / Apple inference friendliness is strategically important.

H. More channels
Examples:
- Discord
- WhatsApp
- Signal
- iMessage
The audit must determine whether this is important now or too early.

I. Lightweight GUI
A lightweight ChatGPT-style local interface is a strong product direction:
- history
- new chat
- visible tools
- visible settings
- model controls
- trust/approval UI

The audit must judge whether this is:
- essential for public V1
- or important but phase-2

J. Sub-agents / orchestration
High-potential differentiator, but also high complexity.
The audit must judge whether it is:
- too early
- or a justified medium-term differentiator

K. Skills / plugin / marketplace direction
The audit must judge whether this is:
- strategically useful later
- or dangerous too early

L. Visual observability / “pixel mode” / fun UI mode
Interesting as branding and observability,
but the audit should likely classify whether this is:
- valuable later
- but not core V1

M. Windows executable / mainstream distribution
The audit should explicitly assess:
- whether architecture is compatible with future .exe packaging
- what would block it
- what would be needed

6. WHAT A CREDIBLE PUBLIC V1 SHOULD PROBABLY DO
==================================================

The audit must evaluate the repo against a practical V1 usefulness bar.

A credible public V1 should likely be able to do at least:
- web research
- know basic local system context
- create/read/edit notes
- inspect and manipulate files with explicit permission
- read basic document formats
- perform simple safe local actions
- preserve useful session continuity
- feel understandable and trustworthy

The audit must decide:
- how far current code is from that
- which missing capabilities are true blockers
- what minimum V1 scope should be
- what should explicitly wait until after V1

==================================================
7. PRIORITIZATION PHILOSOPHY FOR THE AUDIT
==================================================

The audit must not recommend everything at once.

It must clearly separate:
- MUST HAVE for usefulness
- SHOULD HAVE for product credibility
- STRATEGIC DIFFERENTIATORS
- TOO EARLY / SHOULD WAIT
- BAD IDEAS for the current stage

Likely examples:
- system info, notes, files, docs, safe terminal, and simple GUI may be near-term V1 material
- multi-agent orchestration, skills ecosystem, and visual pixel mode may be later-stage differentiators
- broad platform expansion too early may be a strategic mistake

==================================================
8. BOTTOM-LINE DIRECTION TO TEST
==================================================

The audit should explicitly test whether the most realistic and strongest future promise for Unclaw is something like:

**“the simplest, fastest, most understandable local desktop agent runtime for personal machines and open-source models.”**

That means evaluating whether the current repo can realistically evolve toward:
- radical local-first design
- strong privacy posture
- understandable permissioned actions
- small-model friendliness
- low-friction UX
- practical day-to-day usefulness

rather than trying to become:
- a broad gateway-heavy platform
- a giant multi-channel ecosystem too early
- or a feature-bloated clone of OpenClaw
